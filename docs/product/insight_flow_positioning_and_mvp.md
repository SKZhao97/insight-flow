# Insight Flow 定位、亮点与 MVP 设计

## 1. 一句话定位

> 一个面向持续研究任务的 AI-assisted research workflow system，用于把分散信息沉淀为结构化研究输出。

更口语化一点可以写成：

> 用 AI 帮助完成“收集信息 -> 组织证据 -> 形成周报/专题观察”的研究工作流系统。

---

## 2. 项目不是在做什么

为了防止项目失焦，需要先明确它不是什么。

它不是：

- 一个普通 RSS 阅读器
- 一个单篇文章摘要工具
- 一个泛化聊天机器人
- 一个为了堆 LangChain / RAG / Agent 概念而拼出来的 Demo

它真正要做的是：

> 把多源信息处理成带上下文、可复用、可持续积累的研究材料与研究输出。

---

## 3. 核心亮点

### 3.1 亮点一：完整研究链路，而不是单点能力

项目覆盖的是完整链路：

- 信息采集
- 内容标准化
- 去重与筛选
- 单篇分析
- 历史检索
- 报告生成
- 人工修订
- 资产沉淀

这使它具备系统感，而不是功能拼盘。

### 3.2 亮点二：RAG 的使用场景合理

RAG 在这个项目中不是为了做问答，而是为了做历史证据召回。

它服务于：

- 周报生成时补充相关历史内容
- 专题分析时拉取过去的背景材料
- 用已有报告和人工修改增强后续输出

因此，RAG 是研究工作流的一部分，而不是附加演示功能。

### 3.3 亮点三：LangGraph 的使用场景合理

项目并非单线流程，而是一个带状态、分支和人工介入的任务系统。

LangGraph 可以自然承担：

- 流程节点编排
- 中间状态记录
- 条件路由
- 重试或降级处理
- 人工审核后的继续执行

### 3.4 亮点四：Agent 是局部决策，不是全权自治

Agent 最合理的引入方式不是“所有步骤都交给模型自己决定”，而是在关键高不确定节点承担策略判断。

例如：

- Planner：决定当前任务路径
- Reviewer：检查结论是否证据不足或表达过强

这种设计比泛泛而谈“多 Agent 协作”更可信。

### 3.5 亮点五：项目具备长期资产沉淀能力

随着使用增加，系统会积累：

- 原始内容
- 清洗后文本
- 摘要与标签
- 周报与专题稿
- 人工修改记录

这些沉淀物会反过来提升未来任务质量，这也是项目长期可用的关键。

## 4. MVP 边界

MVP 的原则不是“尽量少做”，而是：

> 用尽可能少的产品面功能，跑通尽可能完整的 AI research workflow。

### 4.1 MVP 必须有的能力

#### 输入层

- RSS 源订阅
- URL 输入
- 手动粘贴文本

#### 处理层

- 网页正文提取
- 文本清洗
- 基础元数据抽取
- 基础去重

#### 分析层

- 单篇摘要
- 标签生成
- 分类
- 关键观点提取

#### 检索层

- 基于历史内容的主题检索
- 为周报或专题分析补充上下文

#### Workflow 层

- 用 LangGraph 串起完整任务流
- 至少包含一次条件分支
- 至少包含一次人工介入点
- 至少包含一个 Reviewer 式决策节点

#### 输出层

- 周报生成
- 基础专题分析
- Markdown 导出

### 4.2 MVP 可以不做的能力

- 多用户
- 权限系统
- 支付
- 第三方同步
- 浏览器插件
- 高级任务调度中心
- 复杂多 Agent 拆分
- 图谱可视化
- 精细化运营能力

---

## 5. 建议的最小技术穿插方案

下面这套设计的目标是：让关键概念都出现，但各自职责清晰。

### 5.1 基础 Pipeline

负责稳定处理：

- ingest
- parse
- clean
- dedup
- summarize
- tag
- classify

这一层的目标是把每条内容处理成可存储、可检索、可引用的标准化材料。

### 5.2 RAG 层

负责历史材料召回：

- 按主题检索历史摘要或 chunk
- 按时间窗补充上下文
- 将相关历史材料注入周报/专题生成上下文

这一层的目标是让输出具备连续性，而不是只依赖当前输入。

### 5.3 LangGraph 层

负责编排状态化任务。

建议至少包含以下节点：

- `collect_inputs`
- `normalize_documents`
- `analyze_documents`
- `retrieve_history`
- `draft_report`
- `review_evidence`
- `human_edit`
- `export_markdown`

其中可加入条件分支：

- 如果是单篇处理，则跳过周报生成
- 如果 Reviewer 判断证据不足，则回到 `retrieve_history`
- 如果人工确认通过，则继续导出

### 5.4 Agent 层

先做两个最合理的角色：

- `Planner Agent`
- `Reviewer Agent`

`Planner Agent` 职责：

- 识别当前任务属于单篇摘要、周报生成还是专题分析
- 判断是否需要先做历史检索
- 输出处理策略

`Reviewer Agent` 职责：

- 检查输出中的证据覆盖
- 判断结论是否过强
- 判断是否需要补充历史材料

### 5.5 Multi-agent 演进层

后续如果要扩展，可以再拆：

- `Collector Agent`
- `Analyst Agent`
- `Retriever Agent`
- `Editor Agent`
- `Reviewer Agent`

但第一版不要让多 Agent 成为主角，否则复杂度会上升太快。

## 6. 建议的 MVP 用户场景

第一版不要泛化成“面向所有研究者”，而要聚焦到一类高频、真实场景：

> 持续跟踪 AI / AI Coding / 科技动态，并需要每周形成结构化中文输出的人。

围绕这个场景，用户任务可以设计为三种：

### 6.1 日常采集任务

- 添加 RSS 或链接
- 系统自动抓取、摘要、打标签
- 用户沉淀候选内容

### 6.2 周报任务

- 根据一周内容生成周报草稿
- 自动召回相关历史材料
- 输出主题化、结构化结果
- 用户人工修改后导出

### 6.3 专题研究任务

- 输入一个主题
- Planner 决定是否补充历史检索
- 系统汇总相关材料
- 输出专题观察草稿

## 7. 简历表达反推

如果你希望项目在简历上足够强，MVP 就应该反向服务这些表达点。

目标是最后能自然写出下面这类描述：

- 设计并实现多源内容采集、清洗、去重、结构化分析的 AI research workflow
- 基于 LangGraph 构建状态化任务流，支持条件路由、人工介入和质量审查
- 引入基于历史内容召回的 RAG 机制，增强周报与专题分析的上下文一致性
- 设计 Planner/Reviewer 式 agent 节点，实现检索策略判断与结论质量控制
- 构建可持续积累的个人研究资产库，支持长期复用与持续迭代

如果做完的项目无法自然支撑这些描述，说明 MVP 设计还不够对。

## 8. 建议的一句话卖点

可以从下面几句中选择一个主表述方向：

### 版本 A

> Insight Flow 是一个 AI 驱动的研究工作流系统，用于将分散信息转化为可复用的结构化研究输出。

### 版本 B

> Insight Flow 关注的不是内容摘要，而是“持续研究任务”的信息组织、证据召回与报告生成。

### 版本 C

> 它尝试把 RAG、LangGraph 和 Agent 设计进一个真实可用的研究场景，而不是停留在概念演示。
