# Insight Flow Schema 设计动机详解

## 1. 目的

这份文档不再回答“表是做什么的”，而是回答：

1. 为什么要这样拆表
2. 为什么这些字段要单独存在
3. 为什么某些字段看起来像重复，其实不能省
4. 当前 schema 在为未来哪类问题预留空间

你可以把它理解成：

- `database_schema_design` 讲的是“设计结果”
- `database_table_flow` 讲的是“数据怎么流”
- 本文讲的是“为什么必须这么设计”

---

## 2. 总体设计哲学

Insight Flow 的 schema 不是按照“最少表数”设计的，而是按照“长期研究系统的可追溯性和可演进性”设计的。

这套设计背后有四个核心判断：

1. 内容资产和执行过程不能混在一起
2. 原始事实、结构化理解、最终输出不能混成一层
3. 单次生成不是目标，长期沉淀和复用才是目标
4. 如果未来要排查问题，必须能回到每一层中间结果

所以你现在看到的 schema，会刻意把下面这些东西拆开：

- `Document`
  原始内容资产
- `Summary`
  对内容的结构化理解
- `Cluster`
  对多篇内容的事件级组织
- `WorkflowRun`
  一次系统执行
- `Report`
  最终输出产物

这五类对象虽然都和“周报生成”有关，但它们不是一回事。

---

## 3. 为什么不是“一张大表”

最容易想到的简化方案通常是这样：

- 一张 `documents` 表里既存原文，又存摘要，又存 embedding，又存分类，又存最终报告片段

这个方案的问题很大：

1. 它把“事实层”和“解释层”混在一起
2. 一旦摘要策略变化，就要改原始资产记录
3. 一篇文档可能被多次分析，不同模型和 prompt 结果无法共存
4. 后续做聚类、检索、引用链时会非常混乱
5. 很难回答“错误到底发生在抓取、分析、检索还是生成”

所以 Insight Flow 当前 schema 的一个根本原则就是：

> 一层对象只负责一层语义。

这会让表变多，但换来的是：

- 逻辑清晰
- 可追溯
- 可复算
- 可升级

---

## 4. 为什么 `Source` 和 `Document` 要分开

很多系统会把 RSS URL、文章 URL、手动输入直接都塞进一张内容表里。

Insight Flow 没这么做，是因为：

- `Source` 表示的是“上游入口定义”
- `Document` 表示的是“进入系统后的内容实例”

这两个语义完全不同。

### 4.1 `Source` 的设计动机

`Source` 的关键字段是：

- `type`
- `name`
- `config_json`
- `status`
- `last_synced_at`

为什么需要这些字段：

#### `type`

因为来源层和内容层的处理方式不同。

例如：

- RSS source 需要定时拉取 feed
- 手动 source 可能只是人工输入集合

如果没有 `type`，后续同步逻辑就没有明确路由依据。

#### `config_json`

为什么不用强结构字段？

因为 source 配置天然是异构的。

RSS 可能需要：

- feed url
- polling interval
- include/exclude rules

后续别的 source 可能需要完全不同的配置。

所以在 MVP 阶段用 `JSONB` 比强行定死字段更稳。

#### `status`

为什么来源也要有状态？

因为来源不是“存在就一直有效”。

后面你肯定会遇到：

- 某个 feed 暂停
- 某个 source 质量太差要停掉
- 某个 source 临时不想同步

这类控制不应该依赖删数据，而应该靠状态字段。

### 4.2 `Document` 的设计动机

`Document` 是系统里的原始内容资产。

这里最重要的设计思路是：

> 一篇内容一旦进入系统，就应该先有一个稳定的主对象承载它。

这个对象承载的不是最终价值，而是“所有后续处理的根”。

---

## 5. 为什么 `Document` 里要同时有 `url` 和 `canonical_url`

这两个字段看起来很像，但其实语义不同。

### `url`

表示用户输入的原始 URL，或者抓取时拿到的原始链接。

### `canonical_url`

表示经过归一化之后，系统认为的“规范 URL”。

为什么不能只留一个：

1. 原始输入和标准归一结果不一定一致
2. 很多网站有 tracking 参数、分享参数、移动端参数
3. 去重时应该以 `canonical_url` 为准，而不是原始 URL
4. 排查问题时又必须能看回原始输入是什么

这是一种很典型的“原始值 + 归一值”并存设计。

底层逻辑是：

- `url` 用于还原输入现场
- `canonical_url` 用于业务判重和统一索引

---

## 6. 为什么 `Document` 里要同时有 `raw_content` 和 `cleaned_content`

这是整个 schema 里很关键的一组字段。

### `raw_content`

表示抓取阶段得到的原始文本。

它可能包含：

- 导航
- 广告
- 多余脚注
- 结构噪音

### `cleaned_content`

表示经过 normalization 之后，真正进入后续分析和 embedding 的正文。

为什么不能只保留清洗结果：

1. 清洗规则是会变的
2. 后续你可能会发现某些文章被清洗坏了
3. 排查抓取问题时需要回看原始抽取内容
4. 不保留原始内容，就无法重放清洗过程

所以这对字段的本质是：

- `raw_content` 负责可回放
- `cleaned_content` 负责可处理

这正是“可复盘系统”和“一次性脚本”的分水岭。

---

## 7. 为什么 `Document` 里要有这么多状态字段

很多系统只会放一个 `status`，但这里拆成了：

- `status`
- `quality_status`
- `dedup_status`
- `extraction_method`

这不是冗余，而是在把不同维度的问题分开。

### 7.1 `status`

这是内容处理状态。

例如：

- `ingested`
- `normalized`
- `failed`

它回答的是：

> 这篇文档在处理链路里走到哪一步了？

### 7.2 `quality_status`

这回答的是：

> 这篇内容值不值得继续进入资产链路？

这是价值判断，不是流程状态。

如果把它和 `status` 混在一起，你后面会分不清：

- 是处理失败了
- 还是处理成功但质量太低被拒绝了

### 7.3 `dedup_status`

这回答的是：

> 在相似内容集合里，这篇文档扮演什么角色？

例如：

- 主文档
- 支撑来源
- 重复项

这又是另一维语义。

### 7.4 `extraction_method`

这回答的是：

> 这篇正文是怎么提出来的？

为什么重要：

- 本地提取质量不好时，要知道是不是 fallback 才成功
- 以后比较不同抓取策略效果时，需要有来源依据
- 这也是问题排查的重要维度

所以这里的设计原则是：

> 不同维度的问题，不要被迫挤进同一个字段。

---

## 8. 为什么还需要 `content_hash`

既然有 URL，为什么还需要 `content_hash`？

因为 URL 和内容不是一一稳定对应的。

有几种典型情况：

1. 同一篇内容被多个站转载，URL 完全不同
2. 同一 URL 在不同时间内容会变
3. 某些页面 canonical 不可靠
4. 手动文本输入根本没有 URL

`content_hash` 的作用是：

- 提供一层纯内容视角的去重信号

它不能解决全部重复问题，但它是很便宜、很有效的第一层过滤器。

底层逻辑：

- URL 解决链接身份
- hash 解决文本身份
- 语义相似度解决近重复身份

这三层是互补关系，不是替代关系。

---

## 9. 为什么 `Summary` 要单独成表，而不是挂在 `Document` 上

这是 schema 的另一个关键设计。

如果把摘要、标签、分类这些字段直接塞到 `Document` 上，短期会简单，但中期会出问题。

### 9.1 因为 `Document` 是事实层，`Summary` 是解释层

`Document` 是“这篇东西是什么”。

`Summary` 是“模型如何理解这篇东西”。

这两层在语义上不应该混。

### 9.2 因为同一篇文档可能被多次分析

你已经预留了：

- `prompt_version`
- `model_name`

这意味着未来是允许：

- 同一文档换模型重跑
- 同一文档换 prompt 重跑
- 保留多版分析结果

如果这些字段都塞在 `Document` 上，你就只能覆盖旧结果。

### 9.3 因为检索和报告依赖的是 `Summary`，不是原始 `Document`

后续生成周报时，第一层召回主要依赖：

- 摘要
- 关键点
- 标签
- 分类

这些都属于 `Summary` 层，而不是 `Document` 层。

所以单独建表是更自然的。

---

## 10. 为什么 `Summary` 里要存 `prompt_version` 和 `model_name`

很多项目会省掉这两个字段，只保留“内容结果”。

但 Insight Flow 这个项目不适合省。

原因很直接：

1. 这是 AI 工程系统，不是静态 CMS
2. 模型输出会变化
3. Prompt 设计也会变化
4. 未来你一定会问：
   “这条摘要为什么是这个风格？”

这两个字段的底层价值是：

- 可复现
- 可对比
- 可演进

以后你可以做到：

- 比较 `v1 prompt + model A` 和 `v2 prompt + model B`
- 看哪种摘要质量更高

如果没有这两个字段，所有分析资产都会变成不可解释的黑盒结果。

---

## 11. 为什么同时要有 `SummaryEmbedding` 和 `DocumentChunk`

这看起来也像重复。

但实际上，它们解决的是两种不同层级的检索问题。

### `SummaryEmbedding`

适合回答：

- 哪些内容在主题上相关？
- 最近关于某个方向有哪些值得回看的条目？

它是高层语义召回。

### `DocumentChunk`

适合回答：

- 某个具体结论在原文中是怎么说的？
- 哪一段原始文本能支撑这个观点？

它是低层证据回填。

为什么不能只保留 chunk：

1. 整体召回会很碎
2. 高层主题检索不稳定
3. 报告生成需要先抓大意，不是先抓碎片

为什么不能只保留 summary embedding：

1. 会丢掉原始证据细节
2. 最终输出容易只剩“摘要的摘要”
3. Reviewer 无法回到事实层核查

所以两者共存的本质是：

- `SummaryEmbedding` 负责找方向
- `DocumentChunk` 负责补证据

这是当前项目最关键的 RAG 设计之一。

## 12. 为什么 `DocumentRelation` 必须单独存在

很多人会倾向于在 `Document` 上直接加一个：

- `duplicate_of_document_id`

但这里没有这么做，而是用了单独的关系表。

这是因为文档之间的关系不是单一的。

目前至少有两类：

- `near_duplicate`
- `supporting_source`

未来甚至可以扩展为：

- `translation_of`
- `followup_of`
- `same_topic_as`

如果只用一个外键字段：

- 关系类型表达能力太弱
- 一个文档只能挂一种关系
- 难以保存相似度分数

所以单独建 `document_relations` 的本质是：

> 文档之间的关系，本身也是一种一等公民数据。

---

## 13. 为什么 `Cluster` 也要单独成层

很多简单周报系统会直接拿一堆摘要拼成报告，不做 cluster。

这样做的短期好处是实现简单，但会有三个明显问题：

1. 报告结构会像“文章流水账”
2. 同一事件会被重复写多次
3. 很难形成“这一周发生了哪些关键事件”的自然组织

`Cluster` 的存在，就是把“单篇内容资产”提升成“事件级资产”。

### 为什么 `Cluster` 需要这些字段

#### `title`

需要一个事件级标题，而不是把单篇文章标题直接拿来用。

#### `summary`

需要一个事件级摘要，概括整个簇，而不是照搬单篇摘要。

#### `window_start` / `window_end`

为什么 cluster 需要时间窗？

因为事件聚类不是永恒静态对象，它是在一个时间范围内构建出来的。

同样的主题，下一周可能会形成新的 cluster。

#### `build_version`

因为聚类逻辑以后很可能会变。

如果不保留版本，你将来很难解释：

- 为什么这次聚类结果和上次不同

所以 `Cluster` 其实是“事件视角的可版本化组织结果”。

## 14. 为什么 `WorkflowRun`、`WorkflowEvent`、`RetrievalRecord`、`ContextPack` 都不能省

这是当前 schema 里最偏工程化的一组设计。

如果只看产品表面，会觉得：

- 有了 report 不就够了吗？

但对一个真正要做 RAG、LangGraph、Human-in-the-loop 的系统来说，这远远不够。

### 14.1 `WorkflowRun`

为什么要单独有 run 表？

因为你需要一个稳定对象来承载：

- 这次任务是什么
- 当前状态是什么
- 时间窗是什么
- 可恢复状态是什么

它是整个执行链的主锚点。

### 14.2 `WorkflowEvent`

为什么还要 event 表？

因为 run 只描述整体，不能描述过程。

如果没有 event 表，你没法回答：

- 哪个节点失败了
- 哪个节点被跳过了
- 从哪里恢复的

所以：

- `WorkflowRun` 是“任务级”
- `WorkflowEvent` 是“节点级”

### 14.3 `RetrievalRecord`

为什么检索记录不能只是日志？

因为 RAG 的问题很多时候不是“有没有调用检索”，而是：

- 当时到底检索了什么
- 命中了哪些 summary
- 又回填了哪些 chunk

如果这类信息只在日志里，后续分析会很痛苦。

单独成表之后：

- 可以做回放
- 可以做检索效果分析
- 可以做离线评估

### 14.4 `ContextPack`

为什么最终上下文包也要落表？

因为最终生成质量很大程度取决于：

- 当时到底给模型喂了什么

如果上下文不落地，你只能看到：

- 输入材料存在
- 输出结果也存在

但中间最关键的“拼装结果”消失了。

所以这组表的设计本质是：

> 让生成型系统具备工程系统应有的可诊断性。

---

## 15. 为什么 `Report`、`ReportItem`、`UserEdit` 要分三层

很多简单系统只会有一张 `reports` 表，把最终 Markdown 一存就结束。

Insight Flow 没这么做，是因为这个项目强调：

- 可引用
- 可追溯
- 可编辑

这三件事决定了输出层必须拆开。

### 15.1 `Report`

这是最终产物对象。

它描述的是：

- 一份报告整体是什么

所以它保存：

- 标题
- 时间窗
- 正文
- 状态
- 版本

### 15.2 `ReportItem`

为什么还要 item？

因为报告不是一个不可分割的 blob。

你需要知道：

- 报告里的每一条内容对应了什么证据
- 它属于哪个 cluster
- 它来自哪个 summary 和 document

这就是引用链。

如果没有 `ReportItem`：

- 报告虽然能看
- 但无法精确回溯每个段落的证据来源

### 15.3 `UserEdit`

为什么人工编辑不能只覆盖 `reports.content_md`？

因为编辑本身就是系统的一部分。

你未来很可能需要知道：

- 用户删掉了哪些内容
- 用户补充了哪些内容
- AI 草稿和人工成稿差在哪

所以编辑记录本身也应该被沉淀成资产。

这就是为什么要单独有：

- `before_content`
- `after_content`
- `edit_summary`

## 16. 为什么大量使用 `JSONB`

schema 里不少字段用了 `JSONB`，比如：

- `config_json`
- `key_points`
- `tags`
- `bilingual_terms`
- `state_json`
- `filter_json`
- `score_snapshot`
- `context_json`

为什么不是全部拆成子表？

因为这些字段有一个共同点：

- 结构有约束，但变化频率高
- 访问模式以“整体读写”为主，而不是复杂 join

例如：

### `config_json`

来源配置是异构的，很难一开始完全建模稳定。

### `key_points` / `tags`

它们天然就是小集合结构，用 JSONB 比拆成若干子表更轻。

### `state_json`

workflow state 本身是半结构化对象，强行全部拆表会非常重。

### `context_json`

context pack 往往是一次性拼装后的上下文对象，也更适合整体保存。

这里的设计原则是：

- 核心关系对象用关系表
- 可变的半结构化内容用 JSONB

这是一种典型的 PostgreSQL 混合建模策略。

---

## 17. 为什么大量使用索引和唯一约束

这不是“提前优化”，而是在给未来业务问题设边界。

### 17.1 唯一约束的意义

例如：

- `summaries(document_id, prompt_version, model_name)`
- `summary_embeddings(summary_id, embedding_model, embedding_source)`
- `cluster_items(cluster_id, document_id)`

这些约束的意义是：

- 明确一类结果在什么维度上只能出现一次

如果没有这些约束，后面很容易出现：

- 重复分析结果
- 重复 embedding
- 同一文档反复挂进同一 cluster

### 17.2 索引的意义

例如：

- `documents.published_at`
- `documents.quality_status`
- `documents.dedup_status`
- `workflow_events.node_name`
- `reports.window_start/window_end`

这些索引不是为了早期性能炫技，而是因为后面这些就是高频查询路径。

一个很简单的判断标准是：

> 如果某个字段未来会经常出现在 where、join、排序或者窗口过滤里，就应该在 schema 层正视它。

## 18. 为什么外键删除策略不统一

你会看到有的是：

- `CASCADE`

有的是：

- `SET NULL`

这不是风格不统一，而是对象生命周期不同。

### 适合 `CASCADE` 的

例如：

- `Document` 删除时，`Summary`、`Chunk` 一起删

因为这些对象本来就是从属资产。

### 适合 `SET NULL` 的

例如：

- `Report.generated_by_run_id`

为什么？

因为报告一旦形成，不应该因为 workflow 记录被清理就失效。

所以这里是在表达：

- 哪些对象是强依赖
- 哪些对象只是历史引用

这其实是在用外键策略表达业务生命周期。

---

## 19. 为什么当前 schema 适合 MVP，但又能往后长

当前 schema 没做的事情也很多：

- 没有多租户
- 没有权限系统
- 没有复杂用户画像
- 没有平台化团队协作表

这是刻意的。

因为 MVP 要证明的是：

- 研究资产能不能沉淀
- 工作流能不能闭环
- 报告能不能可追溯地产出

而不是提前证明“平台能力”。

但与此同时，这个 schema 又预留了足够多的演进空间：

- `prompt_version`
- `model_name`
- `build_version`
- `state_json`
- `ContextPack`
- `UserEdit`

这些设计意味着未来你可以往下长：

- 更复杂的分析策略
- 更复杂的 reviewer 逻辑
- 更强的风格学习
- 更丰富的历史对比和复用

所以它的取舍是：

> 不为未来做过度平台化设计，但为未来的 AI 工作流升级预留接口。

---

## 20. 一句话总结这套 schema 为什么这么设计

如果要压成一句话：

> 它不是为了最少表数设计的，而是为了让 Insight Flow 从“生成内容的脚本”长成“可追溯、可复盘、可复用的研究系统”。

再拆开一点就是：

- `Document` 保留事实
- `Summary` 保留理解
- `Cluster` 保留事件组织
- `Workflow*` 保留执行过程
- `Report*` 保留最终产物和引用链
- `UserEdit` 保留人机协作痕迹

这就是当前 schema 的核心设计动机。
